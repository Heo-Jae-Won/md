- **객체지향이란 무엇인가?**
    - **역할과 책임을 갖는 자율적인 객체들의 협력**으로 프로그래밍 하는 것입니다.
    - 객체지향 프로그래밍을 통해 변경에 유연한 코드를 작성할 수 있습니다.
- **역할은 무엇인가?**
    - 역할은 **객체가 협력 안에서 수행하는 책임의 집합** 입니다.
        - **역할은 객체의 추상화** 입니다.
        - **역할을 통해 다형성을 구현할 수 있습니다.**
    - 저는 지금 면접이라는 협력에 면접자라는 역할을 가지고있습니다.
    - 제가 아닌 다른 크루가 면접자라는 역할을 가지고 면접이라는 협력에 참여할 수 있습니다.
- **책임은 무엇인가?**
    - 책임은 **객체가 협력에 참여하기 위해 수행하는 행동**입니다.
    - 저는 지금 면접관님이 보내는 메세지에 답변을 하고 있습니다. 이러한 행동을 수행하는것을 책임이라 말합니다.
- **협력은 무엇인가?**
    - 협력은 **애플리케이션의 기능을 구현하기 위해 수행하는 객체들의 상호작용**입니다.
    - 저희가 지금 면접을 하는 순간도 채용을 위한 협력을 하고 있습니다.
    - 예를들어 저와 면접관님은 면접이라는 협력을 위해 상호작용 하고 있습니다.
    
    - **협력은 어떻게 수행되나?**
        - **객체들의 메세지 교환**을 통해서 이루어집니다.
    - **메세지란 무엇인가?**
        - **객체들이 서로 협력하기 위해 사용할 수 있는 유일한 소통수단**입니다.
        - 메세지는 "어떻게?"가 아닌 "무엇을?"을 표현합니다.
    - **메서드란 무엇인가?**
        - **메세지를 수신했을 때 실제로 실행되는 함수** 또는 프로시저를 메서드 부른다.
- **좋은 코드란 무엇인가?**
    - **변경에 유연한 코드**가 좋은 코드라고 생각합니다.
    - **요구사항은 항상 바뀌고**, 그에따라 **코드도 항상 바뀌어야하기 때문**입니다.
    - **변경에 유연한 코드는 무엇인가?**
        - **협력에 따라 적절하게 추상화**를 해서 **요구사항이 변경**되더라고, **두려움 없이 수정할 수 있는 코드**입니다.
    - **변경에 유연한 코드를 어떻게 작성하는가?**
        - **객체의 상태와 행동을 캡슐화**해서, **변경의 여파를 객체 내부로 제한**하면 됩니다.
        - 그리고 **SOLID**와 같은 설계 원칙들을 지키려고 노력하면 됩니다.
        - 지금 면접이라는 협력에서 면접자라는 역할이 있기 때문에 제가 아닌 다른 크루가 와서 면접자가 되더다도 면접을 수행하는데 문제가 없습니다.
- **객체지향적인 프로그램을 작성하기 위해서 가장 중요하게 생각하는 것은 무엇인가?**
    - 객체의 **상태와 행동을 캡슐화**하는 것이 중요하다 생각합니다.
    - 캡슐화를 하면, 객체 외부에 영향을 주지 않으면서, 객체 내부 구현을 수정할 수 있습니다. 객체 외부는 객체 내부의 세부사항을 모르기 때문입니다.
    - 일급컬렉션의 예를 들 수 있습니다. 컬랙션을 상태로 가지며, 상태 수정에 대한 부분들을 행동으로 제공함으로써, 컬렉션 관리의 책임을 할당할 수 있습니다. (비지니스의 종속적인 자료구조)
- **추상화가 무엇인가?**
    - 추상화는 **불필요한 정보를 생략**하고, 문제 해결에 필요한 **핵심적인 정보만 남기는 작업**이라고 생각합니다.
- **다형성(폴리모피즘)이 무엇인가?**
    - **메세지를 수신한 객체의 타입에 따라 다르게 행동할 수 있는 능력** 입니다.
    - 하나의 인터페이스를 구현하는 여러 클래스들은 각자의 상태에 따라 메서드를 구현할 수 있습니다.
    - 예를들어, MySQL, MariaDB, Oracle사 등은, JDBC API를 자신들의 데이터베이스 엔진에 맞게 구현합니다. 
    - JDBC API를 의존하는 클라이언트는 추후 DB가 변경되더라도 소스 코드를 수정하지 않아도 됩니다.
- **캡슐화란 무엇인가?**
    - 객체의 **상태와 행동을 객체 내부로 숨기는 것**입니다.
    - 이렇게 하면 **결합도는 낮춰지고, 응집도는 높여집니다.**
    - 결국 **변경의 여파를 객체 내부로 제한**할 수 있습니다.
    - 어떻게(상태)를 가리는 방법
- **결합도와 응집도가 무엇인가?**
    - 결합도는 **서로 다른 모듈이 의존하는 정도**를 뜻합니다.
    - 응집도는 **모듈의 내부 요소들이 연관되어있는 정도**를 뜻합니다.
- **결합도를 낮추고 응집도를 높이는 방법은?**
    - **상태와 행동을 객체 내부로 캡슐화**하면 됩니다.
    - 캡슐화를 하면, 객체 내부와 외부의 결합도가 낮아지고, 객체 내부는 연관된 요소들로만 구성됩니다.
    - 추가로 결합도를 낮추기 위해서는 다형성을 구현하면 됩니다.
    - 구현체가 아닌 추상화에 의존하면 다형성을 구현할 수 있는데요, 관련 설계 원칙으로는 의존성 역전 원칙이 있습니다.
    - 인터페이스를 이용해서 의존성 역전원칙을 도입하여, 구현체가 아닌 추상화에 의존하도록 합니다.
- **응집도를 높이면 뭐가 좋나?**
    - 응집도를 높이면, **변경의 대상과 범위가 명확해지기 때문에 코드를 변경하기가 쉬워집니다.**
- **결합도가 낮아지면 뭐가 좋나?**
    - 결합도를 낮추면, **모듈간 의존성이 약해지기 때문에 코드를 변경하기 쉬워집니다.**
    - 예를들어 상태와 행동을 캡슐화한 객체의 내부 구현을 바꿔도, 객체 외부에는 아무런 영향을 주지 않습니다. 객체 외부에서는 객체 내부에 대해서 알 수 없기 때문 입니다.
- **유지보수성은 뭔가?**
    - 유지보수성이란 **두려움 없이, 주저함 없이, 저항감 없이 코드를 변경할 수 있는 능력**을 말합니다.
- **디미터의 법칙은 무엇인가?**
    - **"오직 하나의 도트만 사용하라"**라는 원칙입니다.
    - 객체의 내부 구조에 강하게 결합되지 않도록 협력 경로를 제한하는 원칙입니다.
- **DRY 원칙은 무엇인가?**
    - **"반복하지 마라", "Don't Repeat Yourself"**
    - **동일한 코드를 중복하지 말라**는 원칙힙니다.
- **상속의 장점은 무엇인가?**
    - 중복 코드를 제거하고, 다형성을 구현할 수 있는 장접이 있습니다.
- **상속의 단점은 무엇인가?**
    - 상속은 중복 코드로 제거 용도로만 사용하면, **자식 클래스가 부모 클래스의 캡슐화를 위반**합니다.
    - 예를들어 Stack은 Vector를 상속하기 때문에, Stack은 LIPO 구조임에도 불구하고 원소를 맨 뒤에 넣을 수 있습니다.
- **"상속보다 컴포지션을"에 대해서 설명해달라?**
    - 상속을 통해 코드를 재사용하면, **캡슐화를 깨드리고 변경에 취약**해집니다.
    - 재사용할 객체를 새로 만들 **객체의 상태**로 두어 객체를 조합하는 것을 말합니다.
- **객체지향 프로그래밍이 프로그램을 변경에 유연하고 확장 가능하게 할 수 있는 이유는?**
    - 객체지향의 다형성의 특징이 프로그램을 유연하고 확장 가능하게 한다고 생각합니다.
        - 지금 면접이라는 협력에서 면접자의 역할을 하는 제가 다른 사람으로 변경되더라도, 면접관님께는 아무런 영향이 가지 않습니다.
        - 이는 컴파일 타임 의존성과 런타임 의존성을 다르게 두기 때문에 가능합니다.
        - 이러한 이유로 객체지향 프로그래밍이 프로그램을 변경에 유연하고 확장 가능하게 한다 생각합니다.
- **SOLID가 무엇인지?**
    - 단일 책임 원칙은 **하나의 모듈은 하나의 변경 이유만 가져야 한다**는 설계 원칙입니다.
        - 예를들어 비지니스 로직을 위한 자바 코드와 렌더링을 위한 HTML이 섞여있는 JSP는 단일 책임 원칙을 위반한 사례라고 볼 수 있으며, 이 경우 유지보수성이 매우 안좋습니다.
        - 디스패처 서블릿의 HandlerMapping 인터페이스는 핸들러를 찾는 책임만을 가집니다.
    - 개방-폐쇄 원칙은 **확장에는 열려있고, 변경에는 닫혀 있어야 한다**는 설계 원칙입니다.
    - 리스코프 치환 원칙은 **자식 클래스의 인스턴스는 부모 클래스의 인스턴스를 대체할 수 있어야 한다**는 설계 원칙입니다.
    - 의존관계 역전 원칙은 **상위 모듈과 하위 모듈 둘 다 추상화에 의존해야 한다**는 설계 원칙입니다.
        - 예를들어 디스패처 서블릿은 HandlerAdapter 인터페이스를 통해 다양한 핸들러를 실행시킬 수 있습니다. 😃
        - 먼저 HadnlerAdapter 인터페이스를 구현하는 구체 클래스는 HadnlerAdapter를 대체할 수 있습니다. (LSP)
        - 그리고 디스패처 서블릿과 HandlerAdapter 구체 클래스 둘 다 HandlerAdpater 인터페이스라는 추상화에 의존합니다. (DIP)
        - 결과적으로 디스패처 서블릿을 수정하지 않고도, 새로운 핸들러를 추가할 수 있습니다. (OCP)
    - 인터페이스 분리 원칙은 **클라이언트의 기대에 따라 인터페이스를 분리 해야한다**는 설계 원칙입니다.
        - 자바 컬렉션 프레임워크를 보면 자료구조 별로 인터페이스가 잘 분리되어있음 예시도 들 수 있습니다.
