## <span style="color:#802548">_generics란?_</span>

- 컴파일 시의 타입을 체크하는 기능이다. T, K, V, E 등이 있다.
- 기호의 종류만 다를 뿐 임의의 참조형 타입을 의미하는 것은 같다.

```java
class Box<T> { //제너릭 클래스. T box라고 읽는다. T는 타입 매개변수, Box는 원시 타입이다.
    T item;

    void setItem(T item){
        this.item = item;
    }
    T getItem();
}

Box<String> b = new Box<String>;
b.setItem(new Object());
b.setItem("ABC");
String item = b.getItem; // generics로 Element의 type T를 String으로 정했기 때문에 (String) 형변환 필요 없음.
```

- generics를 쓰는 이유는 runtime에서 오류가 터지기 전에 compile time에서 터뜨리기 위함이다.
- 특히 아래와 같은 Object class를 쓰는 경우, 강제 형변환 시에 runtime에 형이 바뀐다.

```java
class BeverageBox {
	private Object[] beverageBox;

	public BeverageBox(Object[] beverageBox) {
		super();
		this.beverageBox = beverageBox;
	}

	public Object getBeverageBox(int param) {
		return beverageBox[param];
	}

	public void setBeverageBox(Object[] beverageBox) {
		this.beverageBox = beverageBox;
	}
}

class Coffee {}
class Juice {}
```

- 그경우 형변환을 잘못하면 runtime 오류에 빠져버린다.
- 가장 큰 문제는 소스코드 상에서는 빨간 줄도 없어 찾기 어렵다는 점이다.
- Coffee 배열을 가져다 넣었는데, Juice로 형변환을 하려한다. 그런데도 compile 시에 오류가 안난다.

```java
Coffee[] arr = {
		new Coffee(),
		new Coffee()
};
BeverageBox box= new BeverageBox(arr);

Coffee cofee = (Coffee) box.getBeverageBox(0);
Juice juice = (Juice) box.getBeverageBox(1);	//error라는 빨간 줄 출력 안됨.
```

- 하지만 generics를 이용하면 compile time에 이런 일을 미리 방지할 수 있다.
- 아래는 T라는 어떠한 class만 받게 한정한 것이다.

```java
class BeverageBox2<T> {
	private T[] beverage;
	private static int trr;
	
	public BeverageBox2(T[] beverageBox) {
		super();
		this.beverage = beverageBox;
	}

	public T getBeverageBox(int param) {
		return beverage[param];
	}

	public void setBeverageBox(T[] beverageBox) {
		this.beverage = beverageBox;
	}
	
}

class Coffee {}
class Juice {}
```

- T가 Coffee로 정해지면, Coffee와 extends로 묶이지 않은 객체로 형변환이 불가능해진다.

```java
Coffee[] arr = {
	new Coffee(),
	new Coffee()
};
BeverageBox2<Coffee> box2 = new BeverageBox2<Coffee>(arr);
Coffee coffee2 = box2.getBeverageBox(0);
Juice jucie = box2.getBeverageBox(0); //error 라는 빨간 줄 출력됨.
```

- generics는 compile time에 미리 type을 check하여 안정성을 높이기 위한 것이다.
    - static은 T가 들어오기 전에 class type이 명확하게 주어져있어야 한다. 가장 먼저 읽혀 올라가기 떄문이다.
    - generics를 배열로 선언하는 것도 불가능하다. 배열은 runtime에 구체화되기 때문이다.
    - new로 쓰는 것도 불가능하다. new는 runtime에 구체화되기 때문이다.

```java
class Box<T>{
    static T item; //error
    List<String>[] lists = new ArrayList<>();  //error
    T[] itemArr; //ok

    T[] toArray(){
        T[] temArr = new T[itemArr.length]; //error
    }
}
```

## <span style="color:#802548">_generics method란?_</span>

- method의 return type 앞에 붙는 <>generics는 참조변수에 붙는 것과는 완전히 다른 의미다.
- 이 경우, class와 무관하게 method에 별도로 매개변수의 type을 지정하는 의미다.
- genericTest class에 사용된 T, U의 generics type과 method2의 K, V는 서로 별개다.

```java
class GenericTest<T, U> {
	//Class에서 생성자로 가져오는 generics method
	void method1(T t, U u) {
		System.out.println(t + ", " + u);
	}
	
	// class와는 독립적으로 운영되는 generics method
	<K,V> void method2(K k, V v) {
		System.out.println(k + ", " + v);
	}
}


public class GenericMethod {
	
	public static void main(String[] args) {
		GenericTest<String, Integer> gt = new GenericTest<>();
		gt.method1("안녕", 123); //class의 parameter를 사용
		gt.method2(3.14, 999); //class의 것이 아닌 method 자체의 parameter를 사용. 
		
	}

}
```

- generics method를 호출할 때는 아래와 같다.

```java
Generics g = new Generics();
double rect1 = g.<Integer, Double>makeRectangle(p1, p2);
double rect = Generics2.<Integer, Double>makeRectangle(p1, p2); 
```

- generics의 type에 관한 설명인 꺽쇠는 대부분 생략할 수 있다.

```java
Generics g = new Generics();
double rect = Generics2.makeRectangle(p1, p2); 
double rect3 = g.makeRectangle(p1, p2);//generics 꺽쇠는 생략 가능.
```

- makeRectangle의 식은 아래와 같다.

```java
public static <T, V> double makeRectangle(Point<T,V> p1, Point<T,V> p2) {
	double left = ((Number) p1.getX()).doubleValue(); 
	//Number는 Integer, Double의 super class. abstract class
	double right = ((Number) p2.getX()).doubleValue();
	double top = ((Number) p1.getY()).doubleValue();
	double bottom = ((Number) p2.getY()).doubleValue();
	
	double width = right - left;
	double height = bottom - top;
	
	return width * height;
}
```

## <span style="color:#802548">_generics extends, super_</span>

- generics에는 특수한 extends 기능이 있다.
- 아래와 같이 T extends 특정 클래스라면, 특정클래스와 그 아래 자손 class만 generics의 type으로 선언 가능하다.

```java
class FruitBox<T extends Fruit>{
    .
    .
    .

}
```

- 반면에 T super Fruit이라면, 특정클래스와 그 위 조상 class만 generics의 type으로 선언 가능하다.

```java
class FruitBox<T super Fruit>{
    .
    .
    .

}
```

- interface를 implements하는 것일지라도 extends라는 용어를 사용해야만 한다.
- 아래는 &로 묶여서 Fruit과 자손, Eatable과 자손의 교집합의 경우에만 type으로 들어올 수 있음을 의미한다.

```java
class FruitBox<T extends Fruit & Eatable>{
    .
    .
    .

}
```

- 구체적인 예시를 살펴보자.
- Human이 super고 Student는 sub다.
- Room은 T라는 generics를 받는다.

```java
class Human {
	void intro() {
		System.out.println("나 사람");
	}
}

class Student extends Human {
	@Override
	void intro() {
		System.out.println("나 학생");
	}
}

class Room<T> {
	T value;
	Room(T value) {
		this.value = value;
	}
	
	T get()	 {
		return value;
	}
	
	void set(T value) {
		this.value = value;
	}
}
```

- generics method가 여러가지 보인다.
- 처음에 보이는 두 식은 동일한 식이다. 구현 방식이 다를 뿐이다.
- extends 지시어라 super로 올라가지 않기 때문에 runtime 오류에 대비할 필요가 없다.
	- 즉 형검사를 할 필요가 없다는 의미다.


```java
public class Generics5 {
	
	//Human과 그 하위 class만 가능.
	static void printManExtends(Room<? extends Human> room) {
		room.get().intro();
	}
	
	//Human과 그 하위 class만 가능.
	static <T extends Human> void printManExtends1(Room<T> room) {
		room.get().intro();
	}
}
```

- Room class는 generics를 받는다.
- generics는 T로 구현되어 있다.
	- 이 경우 다형성이 적용된다.
	- Human -> Human은 당연히 되어야 한다.
	- 그런데 Human -> Student도 된다. Student는 Human의 sub class기 때문이다.
	- 반대로 Student -> Human은 안 된다. Human은 Student의 sub class가 아니기 때문이다.

```java
public static void main(String[] args) {
	printManExtends(new Room<Human>(new Human()));	  //ok
	printManExtends(new Room<Human>(new Student()));  //ok
	printManExtends1(new Room<Student>(new Human())); //error
}
```

- 이번엔 super 구현을 살펴보도록 하자.
- Room class는 generics를 받는다.
- 여기서는 super도 존재하기 때문에 super는 내가 원하는 instance method가 없을 수 있다.
- 형검사가 필요하다는 의미다. 형검사에 따른 예외도 처리한다.
- generics가 super로 구현되어 있다.
	- Human 자기 자신을 포함한 상위 class만 구현가능하다.
	- 즉 Human과 Object class만 가능하다는 의미다.

```java
public class Generics5 {
	static void printManSuper(Room<? super Human> room) {
		Object obj = room.get();
		
		if (obj instanceof Human) {
			Human h = (Human)obj;
			h.intro();
		} else {
			System.out.println("Human 또는 그 하위 타입이 아닙니다.");
		}
	}
}
```

- 여기서도 다형성이 적용된다.
- human - > human은 된다.
- Object -> Object도 된다.
- Object -> Human도 된다. Human은 Object의 sub class기 때문이다.
	- 그런데 Human -> Object는 안 된다. Object는 Human의 sub class가 아니기 때문이다. 

```java
public static void main(String[] args) {
	printManSuper(new Room<Human>(new Human())); 		//ok
	printManSuper(new Room<Object>(new Object())); 		//ok
	printManSuper(new Room<Object>(new Human())); 		//ok
	printManSuper(new Room<Human>(new Object()));		//error
}
```

- 이번엔 super, extends 없이 whildcard만 존재하는 경우다.
- whildcard는 뭐든 들어올 수 있어서, super처럼 내가 원하는 instance method가 없을 수 있다.
- 형검사가 필요하다는 의미다. 형검사에 따른 예외도 처리한다.

```java
public class Generics5 {
	static void printManWhildCard(Room<?> room) {
		Object obj = room.get();
		
		if (obj instanceof Human) {
			Human h = (Human) obj;
			h.intro();
		} else {
			System.out.println("Human 타입이 아닌데?");
		}
	}
}
```

- whildcard의 경우에도 똑같은 다형성의 원리가 적용된다.
- Object -> Object는 가능하다.
- human -> Student도 가능하다.
- Student -> Human은 불가능하다. Human이 Student의 sub class가 아니기 때문이다.

```java
public static void main(String[] args) {
	printManWhildCard(new Room<Object>(new Object()));		//ok
	printManWhildCard(new Room<Human>(new Student()));		//ok
	printManWhildCard(new Room<Student>(new Human()));		//error
}
```


## <span style="color:#802548">_generics의 whildcard란?_</span>

- 위에서 잠깐 와일드카드를 살펴보았다.
- 와일드카드를 사용하면 한층 자유도가 높아진다. 물론 더 어려워지는 것도 맞다.
- whildcard를 사용한 Collections class의 sort()를 살펴보자.

```java
@SuppressWarnings({"unchecked", "rawtypes"})
public static <T> void sort(List<T> list, Comparator<? super T> c) {
	list.sort(c);
}

@SuppressWarnings("unchecked")
public static <T extends Comparable<? super T>> void sort(List<T> list) {
	list.sort(null);
}
```

- 위와 같은 꺽쇠 T나 <? super T>는 사실 모든 것이 들어갈 수 있다는 의미다.
- 즉 Object와 동일하다. 다만 Object로 쓰지 않는 이유는 형변환 때문이다.
- 제너릭과 원시타입 간의 형변환은 반드시 제너릭의 type이 맞아야 한다.
- 다만 예외적으로 와일드카드만이 type이 달라도 허용된다.

```java
Box<String> box = (Box<String>)new Box<Object> // error
Box<String> box = (Box<String>)new Box<? extends Object> //ok
```

- 이제는 sort()를 살펴보자.

```java
static void sort(List<T> list, Comparator<? super T> comp)
```

- 왜 sort는 매개변수 Comparator의 type을 ? super T로 강제한걸까?
- 그 이유는 유지보수와 재활용성 때문이다.
- 만약 아래와 같이 와일드카드를 쓰지 않았다고 해보자.

```java
static void sort(List<T> list, Comparator<T> comp)
```

- 그렇게 된다면 Apple List를 정렬하기 위해서는 Apple Comparator class가 필요하다는 뜻이다.

```java
class AppleComp implements Comparator<Apple>{
    public int compare(Apple t1, Apple t2){
        return t2.weight - t1.weight;
    }
}

Collections.sort(appleBox.getList(), new AppleComp());
```

- 저기까진 괜찮은데 만약 Grape도 비교해야 한다면 아래와 같은 똑같은 내용의 comp를 또 만들어줘야 한다.

```java
class GrapeComp implements Comparator<Grape>{
    public int compare(Grape t1, Grape t2){
        return t2.weight - t1.weight;
    }
}

Collections.sort(grapeBox.getList(), new GrapeComp());
```

- 매번 비교대상이 늘어날 때마다 이렇게 comparator를 implements한 class를 만들기는 매우 불편하다.
- 따라서 ? super T로 generics를 설정하고, Apple과 Fruit을 sub class로 두는 Fruit class를 만든다.
- 상속관계를 활용한 다형성을 통해 불필요한 중복된 method를 줄이는 것이다.
- 다시말해, 아래와 같은 FruitComp라는 상위 class 하나로 AppleComp와 GrapeComp를 대신한다.

```java
class FruitComp implements Comparator<Fruit>{
    public int compare(Fruit t1, Fruit t2){
        return t1.weight - t2.weight;
    }
}

Collections.sort(appleBox.getList(), new FruitComp());
Collections.sort(grapeBox.getList(), new FruitComp());
```

- 처음부터 Fruit class로 만들면 되지 않냐는 질문이 있을 수도 있다.
- T인 경우, Fruit으로 담으면 전부 담을 수 있지 않냐는 것이다.

```java
static void sort(List<T> list, Comparator<T> comp)

class FruitComp implements Comparator<Fruit>{
    public int compare(Fruit t1, Fruit t2){
        return t1.weight - t2.weight;
    }
}
```

- 하지만 그 경우에는 아래와 같이 Apple만 모으고 싶을 때, Apple만 모으게 compile 시에서부터 강제가 불가능하다.
- Fruit class 모두를 타입 매개변수로 받는 list가 강제되어 apple만 따로, grape만 따로 모을 수가 없다.

```java
List<Fruit> list = new ArrayList<>();
list.add(new Grape("파란", 0));//Fruit이라서 Grape도 담음. 하지만 error가 아님.
list.add(new Apple("빨간",1));//Fruit이라서 Apple도 담음. 하지만 error가 아님.
Collections.sort(list, new FruitComp());
for(Fruit f : list) {
	System.out.println(f.toString());
}
```

- 즉 유지보수에 좋지 않다. 따라서 Comparator class는 generics로 ? super T를 쓰는 것이다.
- Comparator는 FruitComparator를 그대로 유지하면서 온갖 과일을 모두 비교할 수 있다.


```java
static void sort(List<T> list, Comparator<? super T> comp)

List<Apple> appleList = new ArrayList<>();
appleList.add(new Apple("빨강",0));
appleList.add(new Apple("파랑",2));
appleList.add(new Apple("노랑",4));
appleList.add(new Apple("주황",1));

Collections.sort(appleList, new FruitComp());
for(Apple f : appleList) {
	System.out.println(f.toString());
}

List<Grape> grapeList = new ArrayList<>();
grapeList.add(new Apple("빨강",0));
grapeList.add(new Apple("파랑",2));
grapeList.add(new Apple("노랑",4));
grapeList.add(new Apple("주황",1));
Collections.sort(grapeList, new FruitComp());
for(Grape f : grapeList) {
	System.out.println(f.toString());
}
```

- 만약 전부 과일을 다 담은 경우라고 해도 FruitComparator를 활용할 수 있다.
- 그냥 T 타입이었으면 이 모든 Comp를 T에 맞춰 새로 만들어야 했을 것이다.

```java
List<Fruit> fruitList = new ArrayList<>();
grapeList.add(new Apple("빨강",0));
grapeList.add(new Grape("파랑",2));
grapeList.add(new PineApple("노랑",4));
grapeList.add(new Banana("주황",1));
Collections.sort(fruitList, new FruitComp());
for(Fruit f : fruitList) {
	System.out.println(f.toString());
}
```

## <span style="color:#802548">_class의 whildcard란?_</span>

- 매개변수에서 generics를 온갖 것을 다집어넣는 <?>를 준다고 해도, class가 늘 기준이다.
- class가 꺽쇠 T extends Fruit이기 때문에 FruitBox는 Fruit과 자손 class만 들어감을 compiler가 인지한다.
- 그럼 참조변수 box가 FruitBox<? extends Object>라고 해도 그 element인 f는 늘 Fruit와 자손 class만을 갖게 된다.
- 따라서 Fruit f를 받을 수 있다.

```java
class FruitBox<T extends Fruit> extends Box<T>{}

static Juice makeJuice(FruitBox<? extends Object> box){
    String tmp = "";

    for(Fruit f: box.getList()) //ok
        tmp += f + " ";

    return new Juice(tmp);
}
```

## <span style="color:#802548">_이중 generics_</span>


- 마지막으로 그럼 이중 generic을 해석해보자.

```java
public static <T extends Comparable<? super T>> void sort(List<T> list)
//list는 반드시 T타입의 원소를 가져야 한다.
//T는 Comparable interface의 구현체 혹은 상속체여야 한다.
// Comparable은 T 타입과 조상을 대상으로 한다
```

- 아래와 같이 Optional에도 static에 <T>가 붙어있다.
- 그런데 여기는 parameter가 없기 때문에 parameter generics는 아니다.

```java
public static<T> Optional<T> empty() {
    @SuppressWarnings("unchecked")
    Optional<T> t = (Optional<T>) EMPTY;
    return t;
}
Optional<String> optVal = Optional.<String>empty();
Optional<String> optVal = Optional.empty(); //위에서 말했듯 generics의 type에관한 설명인 <String>은 생략 가능.
```
