## <span style="color:#802548">_git 사용자 설정_</span>
- 사용자 설정을 한다.
- git의 유저이름, 이메일과 동일하게 하자.

```sh
git config --global user.name "유저이름" 
#git config --global user.name "Heo JaeWon" 
# 해당 유저이름으로 commit이 남게 설정한다.

git config --global user.email "이메일" 
#git config --global user.email "HeoJaeWon@google.com" 
# 해당 이메일로 commit이 남게 설정한다.

git config --global init.defaultbranch main 
# default branch의 이름을 master가 아닌 main으로 한다
```

## <span style="color:#802548">_git remote 설정_</span>

- clone으로 가져온 repo는 remote가 미리 설정되어 있다.
- 따라서 별도의 remote 설정은 필요없다.
- 혹시 안 되어있다면 아래를 참고해 설정하자.

```sh
git remote add [저장소 alias] [url] :
## 원격저장소를 설정한다. 거의 git remote add origin https://~~~ 형태

git remote -v : 
## 설정된 원격저장소와 저장소이름 정보 출력

git remote set-url [저장소 alias] [url] : 
## 기존 원격저장소를 새로운 원격저장소로 바꿈

git remote remove [저장소 alias] : 
## 원격저장소와 연결 끊으며, 저장소 alias도 사라진다.
```


## <span style="color:#802548">_git 구조 설명_</span>

- untracked -> staging area add -> local repo commit -> remote repo push의 형태
  - https://inpa.tistory.com/entry/GIT-%E2%9A%A1%EF%B8%8F-%EA%B0%9C%EB%85%90-%EC%9B%90%EB%A6%AC-%EC%89%BD%EA%B2%8C%EC%9D%B4%ED%95%B4

- 현재 진행되고 있는 프로젝트 폴더가 working directory
  - 파일에 변화가 생기면 git이 인지할 수 있다.
    - 없는 파일이 추가되었다면 untracked files
    - 있는 파일이 변경되었다면 Changes not staged for commit 상태
    - 모두 git add로 commit 예비 단계에 진입시킴
- commit을 위해 working directory에서 변화된 코드 혹은 생성된 파일이 staging area로 올라간 상태
  - Changes to be committed. 싱테
  - git commit으로 local repo에 확정
- commit을 하게 되면 local repo에 저장된다.
  - 이러한 local repo의 변화된 코드를 remote에 반영하기 위해 push를 날린다.
  - git push로 remote repo에 확정
- remote에 올라온 다른 사람이 한 코드를 받으려면 
  - git pull


## <span style="color:#802548">_local source code 변화 사항마저 다 없애기_</span>
- 셋 중에 하나만 쓰면 remote와 차이가 있는 local repo의 source code가 모두 날라간다.
  - remote에 맞춰서 local을 동기화하기 때문이다.
- untracked 파일의 경우에는 아래처럼 해도 파일이 없어지지 않는다.
  - git이 관리하지 않기 떄문이다.
- local에 가한 변화사항이 필요없다고 판단될 떄만 신중하게 사용하자.

```sh
git checkout HEAD [filename]
git checkout -- [filename]
git restore [filename]
```

## <span style="color:#802548">_git status_</span>
- 현재 나의 repo가 어떤 상황인지 살펴 볼 수 있다.

```sh
git status
```

- 새로 추가된 파일이 있다면 

```sh
Untracked files:
  (use "git add <file>..." to include in what will be committed)
        dd.jpg
```

- 아무런 변화가 없다면 

```sh
nothing to commit, working tree clean
```

- 변화가 있지만, git add를 수행하지 않았다면

```sh
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
        modified:   README.md
```

- 변화가 있고, git add를 수행했다면

```sh
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
        modified:   README.md
```

- 변화가 있고, git add를 수행하고, git commit까지 마친 경우

```sh
nothing to commit, working tree clean
```

- pull 혹은 revert 시 conflict가 있는 경우

```sh
Unmerged paths:
  (use "git restore --staged <file>..." to unstage)
  (use "git add/rm <file>..." as appropriate to mark resolution)
        deleted by them: README.md
```


## <span style="color:#802548">_git add_</span>

- commit을 하기 전의 준비 단계. 

```sh
git add [file이름] 

git add . : 
## 수정, 삭제, 추가된 파일들을 전부 staging area에 반영한다. git ignore에 있는 파일은 제외된다.

git restore --staged . : 
## git add한 것을 모두 staging area에서 제외시킨다.
## local의 소스코드가 사라지는 것은 아니다.
```

## <span style="color:#802548">_git commit_</span>

- local repo에 여태까지 해 둔 파일들의 이력을 저장함
- git commit으로 그냥 여는 것은 추천하지 않음.
  - vim이라는 linux의 문서 편집기에 대해서 사전지식이 필요.
  - i를 누르고 commit message를 쓴 뒤 esc 후 :wq를 쓰면 커밋됨
- git commit -m "message" 형태 혹은 IDE와 연동된 GUI를 권장. vscode 권장..

```sh
git commiit # git bash에서 commit을 하게 되면 vim 형태로 열려서 비추천

git commit -m "message"

git reset --soft : 
## 커밋이력을 없애면서 여태까지 git add한 것을 보존하면서 소스코드는 유지된다. 
## 커밋 이후에 뭔가 더 추가해야한다면 자주 쓴다. git reset --soft + git add . + git commit -m "message"의 형태다. 
```


## <span style="color:#802548">_remote에 아직 안 올렸고, 잘못 commit한 경우_</span>
- 아직 remote에 올리지 않은 commit을 전제로 한다.
- 이미 remote에 올라간 파일은 아래와 같이 수정할 수 없다.
  - commit id가 바뀌기 때문에 conflict의 원인이 되기 때문이다.

- 하다 보면 commit에 내용을 더 추가해야 하거나, 오류가 있는 경우가 있다.
  - 그래서 수정한 경우 반드시 새로운 commit을 만들 필요가 없다.
  - 아래와 같이 수정이 가능하다. 
  - commit message를 수정하고 싶을 때도 아래와 같이 사용할 수 있다.

```sh
git add [고친파일명]
git commit --amend
```

## <span style="color:#802548">_git pull_</span>
- remote에 있는 code와 local에 있는 code를 동기화시키는 명령어다.

```sh
git pull
```

- commit 올리기 전에 git pull을 하고 올리자.
- git pull을 학원에 와서, 점심 먹고 나서, 가기 전에 3번은 최소로 실행하자


## <span style="color:#802548">_remote에 올렸는데, 잘못 commit한 경우_</span>

- 이 경우는 해당 commit을 무효화하는 또 다른 commit을 해야된다. 그게 revert다. 
- commmit hash 값을 영역지정해 복사할 때, ctrl + c가 아니라 shift + insert로만 복사할 수 있다.
  - git bash는 linux의 단축키 설정을 갖고 있기 때문이다.

```sh
git revert --no-commit [되돌아갈commit-hash값]...[HEAD commit hash값] : 
## 여러 개의 commit이력을 없애고자 할 때 전부에 대해 revert commit을 남기지 않고 1개의 commit만 남긴다.

git revert --continue

git push

git revert --abort
#아예 revert 행위를 되돌리고 싶다면
```

## <span style="color:#802548">_git conflict_</span>
- conflict 해결의 대원칙은 아래와 같다.

```
1. 업무가 아닌 함수 단위 commit
2. 하루 3번 원격 저장소 동기화 - 출근 직후, 점심 먹고, 퇴근하기 1시간 전
```

- conflict가 나면 아래와 같이 나타난다.

```sh
# <<<< ---> 이전에 있던 소스
# ===
# >>>> --> 새롭게 가져온 소스
```

- conflict는 아래와 같이 해결해준다.

```sh
git status : 
## unmerged된 파일이 무엇인지 살펴본다.

git diff : 
## git status로 확인한 문제되는 파일에서 어떤 부분이 문제인지 살펴보고 원하는 방향으로 정리한다.
## 이 명령어를 bash에서 쓰는 것은 추천되지 않는다. vscode에서 살펴보는게 훨씬 편하다

git add . : 
## 수정한 사항을 staging area에 넣는다. unmerged의 경우, 수정하지 않고 그냥 git add . 해도 수정된 것처럼 인식된다. 따라서 수정을 했으면 무조건 파일을 저장하고 git add를 해야한다.

git merge --continue : 
## merge를 완료하는 commit을 남기면서 conflict를 해결한다.
```

## <span style="color:#802548">_git log_</span>
- git의 commit history를 확인하는 데 필요한 명령어

```sh
git log
#가장 기본이 되는 로그
# q를 눌러서 빠져나온다.
# 아래로 내리려면 화살표 아래를 클릭하여 로그를 확인한다.

git log --oneline 
# commit 로그를 짧게 중요한 정보만 요약해 출력함.

git log --since="2023-04-02" 
# 해당 날짜 이후의 commit 이력만 출력한다. 

git log --author=username 
# 해당 username이 남긴 commit 이력을 확인한다.
```

## <span style="color:#802548">_git stash_</span>
- 변화를 가지고 git switch branch하면 되는 경우도 있다.
  - git add를 안해도 되는 경우는, conflict가 없는 경우다.
  - 이동하려는 branch가 fast-forward가 가능하다면 git add를 하지 않아도 branch switch가 가능하다.

- 만약 conflict가 일어날 것이 분명하다면, git switch가 불가능하다.
- 그럴 때 git stash를 사용해 local changes를 저장한다.

- 다만 버그를 해결하면 git stash clear를 꼭 해줘야 한다.
  - 그렇지 않으면 자신의 commit history graph에 stash도 남아있어 굉장히 지저분해진다.


```sh
git stash 
## 작업 파일 임시 저장

git stash apply 
## feature/insert branch에 임시로 저장해둔 작업 파일을 꺼냄. 

git stash drop stash@{0} 
## 0번째 index stash 삭제

git stash list 
## 임시저장 파일 목록 확인

git stash clear 
## 전체 stash를 다 비운다
```
