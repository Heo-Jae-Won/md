## <span style="color:#802548">_1.언어 문법을 외워야 한다_</span>
- 해당 언어에 관해 잘 알아야 한다.
- 언어에 관해 잘 몰라서 매번 구글에 api를 물어보면서 코드를 만들면 작업 중단이 자주 일어나는 것이라서 업무 집중도가 떨어지게 된다.
  - (1) api는 '꾸준히' 학습해야 한다. 플래시카드도 좋다
  - (2) api는 '처음' 배울 때는 빡세게 학습한다. 교과서의 예제를 백지에서 써버릇해야 한다.
  - (3) api는 한번 익히고 나서는 '1달'에 한번  정도 기본 문법을 상기시킨다
  - (4) api의 활용 수준을 높이려면 다른 사람이 만든 활용 예제를 자주 보고, 이를 백지에서 써버릇해야 한다
  - (5) 다른 언어도 알고 있다면, 다른 언어와 비교해가면서 공통점과 차이점을 떠올린다
  - (6) 해당 언어의 새로운 점을 배울 때는 이전의 가정을 버리려고 노력해야 한다.
    - sync하게 쓰는 java와 async하게 쓰는 java는 완전히 다른 방식으로 작동한다.
    - sync한 방식처럼 debugging하려고 하면 안된다.
  - (7) 처음에 언어를 비유적으로 배우더라도, 나중에는 구체적인 컴퓨터 작동 원리에 의거한 이해로 변경해야한다. 
    - 추상화 수준을 내려서 구체적인 수준의 이해를 해야한다. 물론 쓰는 건 비유적으로 쓰는 추상화 수준을 유지해야만 한다. 사람이 이해할 수 있게 써야 한다.
    - 아래의 소스코드를 보자.
```java
double celsius = 10;
double fahrenheit = (9.0 / 5.0) * celsius + 32;
```

- 우리가 이해하는 방식은 수학에서의 방식이다.
```java
double fahrenheit = ((9.0 / 5.0) * 10 ) + 32;
```

- 그러나 실제 일어나는 작업은 stack에 9.0/5.0을 나눈 결과를 넣고, 
- 다시 스택에서 꺼내 10을 곱하고, 
- 다시 그 결과를 stack에 넣고, stack에서 꺼내서 32를 더한다.
- 하지만 쓰는 방식은 위처럼 써주는 것이 훨씬 바람직하다. 
- 사람이 읽을 수 있게 코드를 써야한다.
<br>

- 하지만 프로그래머는 코드를 보고 추상적인 수준에서 구체적인 수준으로 내려갈 수도 있어야 한다.
  - x라는 변수를 처음에는 상자, 라벨로 이해할 수 있다. 
  - 하지만 실제로 기계적 수준에서 x라는 변수는 상자도, 라벨도 아니다. 
  - 특정 메모리 주소가 담고 있는 값에 대한 참조일 뿐이다.

## <span style="color:#802548">_2.코드를 잘 분석하는 방법_</span>
- (1) 처음 읽을 때는 코드를 프린트해서 읽는다.
- 우선은 코드를 자세히 읽기보단 개략적으로 훑어보며 process를 파악하기 위해 노력한다.
- process step 단위로 쓰인 chunk 해설 comment가 있는지 살펴본다
- 모르는 도메인 변수명이 있다면 표시해두고 history를 찾아보거나 해당 프로젝트 내에서 검색해본다
- 그렇게 파악한 것은 process step 별로 line 시작 statement를 남긴다.

- (2) 변수명, 클래스명, method명에 집중하자
- 3가지 표식은 해당 소스코드에 관한 많은 정보를 제공한다
- 그러나 믿으면 안된다. 농협에서 난 변수명과 클래스명에 자주 데여봤다.
  - mnAddInfo()라는 method가 카드번호 암호화를 수행하는 method였다. 시발 이게 말이되냐?


- (3) 따로 빼 둔 method가 이해가 안되면 따로 빼 둔 method를 inline으로 바꿔 흐름을 파악한다
- 흐름 파악을 완료하면 다시 revert시키고, 해당 method의 역할을 주석으로 남긴다.
